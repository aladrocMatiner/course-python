# Course-level audit workflow

## Build the declared learning graph

1. Inventory every unit, route, checkpoint, companion project, and capstone.
2. Give each required concept and observable checkpoint a stable identifier.
3. Add directed prerequisite edges only when the learner must already know the source concept to complete the target.
4. Distinguish required edges from optional previews and enrichment links.
5. Detect cycles, unknown concepts, forward required dependencies, missing entry concepts, and checkpoints with no observable assessment.
6. Compare the declared graph with actual examples, exercises, hints, and solutions; automation cannot prove that the declaration is true.

Do not rename stable public chapter paths merely to make numbering look sequential. Publish an alternate route map when the safe pedagogical order differs from the numerical catalogue.

## Define routes

For each essential, intermediate/professional, advanced, and optional hero route, state:

- intended learner and prior knowledge;
- exact unit/subsection sequence;
- estimated sessions or time range;
- runnable or observable outcome;
- independently completable checkpoint;
- exit criteria and a safe stopping point;
- optional previews that can be skipped without blocking required work.

Keep the essential route calm and standard-library-first where practical. Do not make advanced setup or theory a hidden prerequisite for an earlier checkpoint.

## Audit cognitive load

Flag a learning step when it introduces several unrelated concepts, unexplained jargon, a new toolchain plus a new programming model, or a solution that adds concepts absent from the prompt. Split the step, move depth behind a route boundary, add a glossary, or reuse a familiar project.

Assess cognitive load from actual learner actions, not chapter length alone. Preserve useful advanced material when it can be made optional and locally signposted.

## Trace assessment and capstones

For every objective, trace:

`objective → explanation → prediction → runnable/observed example → guided change → boundary/error recovery → explained solution → checkpoint/rubric`

Require at least one observable assessment for each required objective. Reuse earlier artifacts in a longitudinal capstone; do not introduce several unrelated technologies only at the final step.

Map capstone features back to prior checkpoints and identify any concept that appears for the first time in assessment.

## Report

Provide graph defects, route entry/exit contracts, cognitive-load findings, traceability gaps, and proposed reversible edits. Separate machine-detected graph inconsistencies from professor judgment and from evidence that still needs learner testing or competent human review.
